# archive/ — تخطيط تاريخي متجاوَز

الملفات هنا **لا تُعتمد** كمرجع حالي. هي التخطيط الأصلي الذي أنتج النظام، محفوظ للسياق التاريخي فقط (كيف تطوّر الفهم، ولماذا اتُّخذ كل قرار في حينه).

**المرجع الحي الحالي هو `plans/reference/`** — كل قرار موثّق هنا استُخلص محتواه (مع أي تصحيح لازم) إلى الملف المناسب هناك. عند أي تعارض بين هذه الملفات و`reference/` أو الكود الفعلي: **الكود يفوز أولاً، ثم `reference/`**.

## المحتويات

- `v1/` — الجولة الأولى الكاملة من التخطيط قبل التنفيذ:
  - `00-instructions-for-claude-code.md` — تعليمات البدء الأصلية لأداة التخطيط.
  - `01-context.md` … `04-open-questions.md` — السياق، نموذج البيانات، تسلسل العمل، النقاط المعلّقة كما كانت *قبل* التنفيذ.
  - `05-plan-update-1.md` — دليل تعديلات جوهرية (علاقة 1:N بدل 1:1، توقيت توليد النشاط، عزل رؤية المراقب، تفاصيل الرفض) طُبِّقت لاحقاً مباشرة داخل 01–04.
  - `raqib-master-reference.md` — محاولة توحيد الملفات أعلاه في مرجع واحد، أُنتجت أثناء التخطيط قبل بدء التنفيذ الفعلي بـ Cursor.
- `raqib-verification-prompt.md` — طلب تدقيق (prompt) لتتبّع سيناريو استخدام حقيقي عبر الكود مقارنة بـ `raqib-system-reference.md` القديم. أداة تشخيص لمرة واحدة، وليست مرجعاً دائماً.
- `raqib-system-reference.md` — "المرجع الموحّد النهائي" الذي كُتب بعد `v1/` ليكون مرجعاً وحيداً، لكنه بدوره صار متجاوَزاً بتطوّر التنفيذ الفعلي (سُمِّي حرفياً "لا تعتمد على أي ملف سابق غير هذا" — وهذا بالضبط ما تسبّب بتضارب المراجع الذي دفع لإعادة التنظيم هذه).

## لماذا لم تُحذف هذه الملفات؟

لأنها توثّق *رحلة القرارات* (لماذا اعتُبرت العلاقة 1:1 ثم صُحِّحت إلى 1:N، لماذا فُصل "المصدر" عن "النشاط"، إلخ) — سياق مفيد لفهم "لماذا البنية الحالية هي هذه" حتى لو كانت التفاصيل التقنية فيها قديمة الآن.
