# raqib — من Excel إلى نظام ويب: المرجع الموحّد الكامل

> ملف واحد شامل. لا تعتمد على أي ملف تخطيط سابق غير هذا — هذا يلغيهم ويجمعهم. اقرأه كاملاً قبل أي قرار تصميم أو كود.

## 1. السياق

جمعية خيرية (غزة) عندها نظام رقابة وتقييم أداء (M&E/KPI) مبني بالكامل على Excel (12 ورقة، معادلات مترابطة). المطلوب: تحويله لتطبيق ويب — **Laravel 11 + Blade + Bootstrap + MySQL**، عربي RTL.

**الفكرة الأصلية بالنظام:** مراقب ميداني يزور أقسام/مشاريع الجمعية، يسجّل نشاط رقابي، النظام يحسب KPI بمعادلة مرجّحة ثابتة.

**كود موجود مسبقاً (لا تصطدم فيه):**
- `users` — موجود.
- `constants` — key/value json، جاهز لتخزين الثوابت (سلالم، طريقة/مرحلة المراقبة، خيارات checklist).
- نظام صلاحيات = **abilities array بملف PHP** (ليس roles تقليدية، رغم وجود جدول `role_user` غير مستخدم فعلياً بهذا النمط). صمّم صلاحيات raqib ضمن هذا النمط.
- `activity_logs` — audit trail عام للمستخدمين، **لا علاقة له بجدول النشاطات الرقابية الجديد**. تجنّب أي تسمية تلتبس معه (اقترح اسماً بديلاً واضحاً لجدول النشاطات، مثل `monitoring_activities`).
- `currency` — موجود، غير مستخدم في raqib.
- Auth + Policies جاهزة.
- لا حاجة لـ Git — يُدار يدوياً.

**ملغى من النطاق:** استيراد بيانات Excel القديمة. العميل سيُدخل بيانات جديدة نظيفة يدوياً.

**أولوية التنفيذ (بالترتيب، لا تحد عنه):**
1. الأساس: DB + ثوابت + جدول الأشخاص + صلاحيات.
2. جدول **النشاطات** (الكيان المركزي).
3. **المشروع** (جدول + فورم + دورتا الاعتماد والمراقبة).

**مؤجَّل تماماً (لا تخطط له بتفصيل الآن):** محضر الاجتماع، تقرير القسم، كل التقارير التحليلية المشتقة.

## 2. القرار المعماري الجوهري: فصل "المصدر" عن "النشاط"

كان الفهم الأول: كل شيء (مشروع، محضر، نشاط خارجي) يُحشر بجدول `DATA` واحد بنفس الحقول — طابق Excel حرفياً لكنه غلط لتطبيق ويب حقيقي.

**القرار النهائي:**
```
مصادر (كل واحد جدول مستقل بحقوله الخاصة):
  projects (مشاريع) ── تحتاج تعديل/دخول متكرر، تقرير غني خاص بها
  meetings (محاضر) ── مؤجَّل التنفيذ
  departments (أقسام) ── مؤجَّل التنفيذ

          ↓ كل مصدر "يولّد" نشاط/أنشطة ↓

  activities (النشاطات) ← الكيان المركزي المُراقَب
     • يحمل الحقول المشتركة لأي نشاط (هرم تنظيمي، مراقب، تقييم KPI...)
     • هو من يمرّ بمراحل المراقبة — ليس المصدر
     • مربوط بمصدره عبر source_type + source_reference
```

**نشاط خارجي مباشر** (مثال: متابعة سيارة، متابعة مبنى): **لا يحتاج جدول مصدر منفصل** — يُدخل مباشرة في جدول النشاطات نفسه، بنوع مصدر مختلف (`source_type = external`)، بدون `source_reference`.

---

## 3. علاقة المشروع بالنشاط: 1:N (نقطة حرجة، تم تصحيحها من قرار أقدم كان 1:1)

- كل مشروع يولّد **نشاطاً أساسياً (primary)** واحداً إلزامياً.
- يمكن أن يولّد **أنشطة تابعة (secondary)** لاحقاً، عددها غير محدود، تُنشأ يدوياً (مثال مستقبلي: تحويل ملاحظة ميدانية لنشاط مستقل بضغطة زر — **لا تبنِ هذا الجزء التنفيذي الآن**، فقط البنية يجب أن تدعمه).
- **الحقول اللازمة على النشاط:** `source_reference` (FK للمشروع، **بدون unique constraint** — عدة أنشطة تشترك نفس المرجع)، و`activity_role` (enum: primary/secondary).
- كل الأنشطة (أساسية وتابعة) لنفس المشروع تمر كل واحدة **بسلسلة مراقبة مستقلة خاصة بها**.

## 4. الجدولان المركزيان

### 4.1 جدول النشاطات (اسم مقترح: `monitoring_activities` — تجنّب `activities` لو وجدت التباس محتمل، وتجنّب `activity_logs` قطعياً)

| الحقل | ملاحظات |
|---|---|
| رمز النشاط | تلقائي، بادئة حسب نوع المصدر (مثال: مشروع/محضر/خارجي بادئات مختلفة — دقة البادئات مفتوحة)، **قابل للتعديل يدوياً، ممنوع التكرار** (unique constraint فعلي) |
| source_type | مشروع / خارجي / (لاحقاً: محضر / قسم) |
| source_reference | FK اختياري (فارغ للخارجي)، **بدون unique** — تعدد أنشطة لنفس المصدر مسموح |
| activity_role | primary / secondary |
| المركز، الدائرة، القسم | هرم تنظيمي 3 مستويات (dependent dropdowns) |
| المسؤول عن النشاط | FK إلى `people` |
| المراقب | FK إلى `people` (يجب أن يملك حساب `user`) |
| التاريخ، الوقت | إدخال |
| اليوم، الشهر، السنة | **محسوبة تلقائياً** من التاريخ |
| نوع النشاط | FK لثابت (10 قيم من Excel: تفتيش ميداني، جودة خدمة، فحص سلامة، فحص مخزون، متابعة شكاوى، مراجعة إجراءات، جرد مفاجئ، مراجعة عقود، تدقيق مالي، متابعة حضور) |
| الممول | FK إلى `funders` |
| حقول بيانات المشروع (مدير المشروع، عدد المستفيدين، مناطق التنفيذ، الميزانية، نسبة/حالة الجاهزية، التوصيات) | **كل هذه اختيارية على مستوى النشاط** — تظهر فقط لو source_type=مشروع، والنشاط يُسمح أن يُحفظ ناقصاً منها |
| الموضوع، الملاحظة، مشكلة ميدانية (bool)، الإجراء المتخذ | إدخال حر |
| قيمة التنفيذ (رقم) | تُؤخذ من نسبة الجاهزية **بعمود المراقب فقط** في تقرير المشروع، وليست إلزامية التلقائية بالكامل (تعديل يدوي ممكن من مدير الرقابة/الإدارة العامة). يمكن أن تكون صفراً مؤقتاً |
| قيمة الجودة، الإغلاق، الخصم (أرقام) | تُدخَل يدوياً من الإدارة العامة/مدير الرقابة فقط |
| KPI | `(تنفيذ×0.4)+(جودة×0.3)+(إغلاق×0.3)−خصم` — محسوب |
| التصنيف | مشتق من KPI: ≥98 ممتاز جداً / ≥90 ممتاز / ≥75 جيد / ≥60 مقبول / ≥40 ضعيف / <40 خطر شديد |
| تسمية كل تقييم (4) | **مشتقة من الرقم للعرض فقط**، غير مخزَّنة كمصدر — القاعدة: رقم دائماً هو المصدر، التسمية عرض |
| ✓ تحقق | **محسوب فقط (لا يُدخل يدوياً، لا يمنع الحفظ)** — يفحص: صحة الهرم، صحة القوائم، تناقض خصم/مشكلة، منطق الإغلاق، نواقص الحقول حسب source_type |
| طريقة المراقبة، مرحلة المراقبة | يحددهما **مدير الرقابة العامة فقط**. القيم الحالية من Excel: "ميداني" و"أثناء التنفيذ" فقط (غير نهائية — انظر القسم 7) |
| حالة المراقبة (workflow status) | بانتظار مراقب / قيد المراقبة / بانتظار تأكيد نهائي / مكتمل المرور |
| اكتمال المرور | **حقل مستقل عن اكتمال البيانات** — يُؤكَّد حتى لو الحقول/KPI ناقصة |

**صلاحية التعديل بعد الإنشاء:** محصورة بمدير الرقابة العامة + الإدارة العامة فقط (عبر abilities).

### 4.2 جدول المشاريع (`projects`)

| القسم | الحقول |
|---|---|
| بيانات المشروع | اسم، رقم (رقم صافٍ — صيغته النهائية غير محسومة، انظر القسم 7)، نوع، الممول، مندوب مشتريات، مدير المشروع (FK `people`)، منسق المشروع (FK `people`، **قد يكون نفس مدير المشروع**)، تواريخ التنفيذ، الموقع |
| بيانات التنفيذ | إجمالي المستفيدين، عدد مناطق التنفيذ، المدة، الميزانية |
| بيانات المراقب الميداني | اسم المراقب، طريقة/مرحلة المراقبة، تاريخ المراقبة (تُملأ في مرحلة المراقبة) |
| قائمة التحقق من الجاهزية | **ديناميكية بالكامل** (مجموعات + بنود)، التحكم بالديناميكية **محصور بأدمن النظام فقط** عبر لوحة تحكم. المجموعات الحالية من Excel (نقطة بداية لا قائمة نهائية): اللوجستيك والموارد، التحضيرات الميدانية، الموارد البشرية. كل بند: عمودان (منسق + مراقب)، خيارات: جاهز/غير جاهز/غير مطلوب. **"غير مطلوب" يُستثنى من المعادلة تماماً** |
| ملخص الجاهزية (محسوب) | نسبة مستقلة لكل مجموعة (منسق ومراقب منفصلين) → تقييم إجمالي. **نسبة المراقب الإجمالية = قيمة التنفيذ في KPI**. نسبة المنسق توثيقية/مقارنة فقط |
| ملاحظات وتوصيات | قوائم نصية حرة من المراقب الميداني |
| الاعتماد والتوقيع | توثيق تأكيدات المراقب، مدير المشروع، مدير الرقابة |
| حالة المشروع (workflow) | بانتظار المنسق / بانتظار مدير الدائرة / بانتظار مدير الرقابة / تم تعيين مراقب (انتقل لسلسلة المراقبة) |
| حقول الرفض | سبب الرفض (نص إلزامي عند الرفض)، من رفض (FK)، تاريخ الرفض، **مسؤولية النقص** (قيمة من قائمة قصيرة قابلة للتوسيع: منسق/مدير مشروع/غير محدد — وليست نصاً حراً) |

**قاعدة عرض بيانات مهمة:** المراقب **لا يرى بيانات أو هوية المنسق ولا تعبئته** بقائمة التحقق. الاستعلام/الواجهة الخاصة به تعرض فقط بيانات المشروع الفعلية + عموده هو بالتحقق.

### 4.3 جدول الأشخاص (`people`)

جدول موحّد — **ليس** `users` منفصل بازدواجية كاملة. كل شخص (مسؤول نشاط، منسق، مدير...) سجل بـ `people`، مع `user_id` **اختياري (nullable)** لو له حساب دخول فعلي في `users` الموجود.

**قاعدة صارمة:** أي اختيار شخص بالنظام = انتقاء من `people` (قائمة/بحث)، **ليس** نص حر إطلاقاً.

شخص واحد قد يحمل أكثر من دور وظيفي (مثال مؤكد: مدير المشروع يعيّن نفسه منسقاً لنفس المشروع).

### 4.4 الهرم التنظيمي

3 مستويات مرتبطة (dependent dropdowns): **المركز** (2: الجمعية / المراكز الصحية) → **الدائرة** (~6-9 بالجمعية + دوائر/مستشفيات بالمراكز الصحية) → **القسم** (~66 إجمالاً، كل قسم تابع لدائرة واحدة فقط).

### 4.5 الثوابت — استخدم جدول `constants` الموجود (key/value json)، لا تنشئ جداول lookup منفصلة لهذه:

**السلالم الأربعة** (رقم دائماً هو المصدر، التسمية للعرض):
- التنفيذ: ممتاز=100 / جيد=80 / مقبول=60 / خطر=40
- الجودة: ممتاز=100 / جيد=85 / مقبول=70 / ضعيف=50
- الإغلاق: مكتمل=100 / جزئي=60 / معلّق=30 / مفتوح=0
- الخصم: لا خصم=0 / تأخير=−5 / عجز=−10 / جودة=−15 / امتثال=−20 / مخالفة=−25

**خيارات بند قائمة التحقق:** جاهز / غير جاهز / غير مطلوب.

**الممولون (`funders`):** جدول منفصل بسيط — الاسم فقط حالياً (قابل للتوسعة لاحقاً بلا قيود تصميمية).

## 5. الأدوار والصلاحيات

**الأدوار التشغيلية (ضمن النطاق الحالي):** مدير مشاريع، منسق، مدير دائرة، مدير الرقابة العامة، مراقب.
**خارج سلسلة الاعتماد لكن موجودة:** الإدارة العامة (يرى كل شيء، ليس جزءاً من الموافقات)، أدمن النظام (صلاحية كاملة + إدارة الصلاحيات والثوابت).
**مؤجَّل:** مسؤول القسم كدور مستقل.
المراقبون **لا يرتبطون بأقسام/دوائر** — كلهم تابعون لدائرة الرقابة العامة مباشرة تحت مدير الرقابة (فريق واحد، يوزَّع حسب رؤيته).

صمّم كل هذا كـ **abilities** ضمن النمط الموجود بالكود (ملف PHP array)، وليس نظام roles جديد.

## 6. تسلسل العمل (Workflows) — سلسلتان منفصلتان تماماً، متعاقبتان

### السلسلة 1: تحضير/اعتماد المشروع (إدارية بحتة — تخص المشروع، لا علاقة لها بالمراقبة)

1. **مدير المشروع** ينشئ المشروع، يملأ بياناته وبيانات التنفيذ، يحدد **المنسق** (قد يكون نفسه).
2. **المنسق** يعبّئ عموده بقائمة التحقق:
   - له حساب دخول ومختلف عن مدير المشروع → يعبّئ بنفسه، يرسل بزر صريح.
   - بدون حساب دخول → **مدير المشروع يعبّئ نيابة عنه**.
   - حتى لو له حساب: يُفضَّل توفير خيار لمدير المشروع بالتعبئة نيابة عنه أيضاً (مرونة).
   - **مدير المشروع = المنسق نفسه** → تُتخطّى خطوة الإرسال/الموافقة المنفصلة تماماً، يكمل مباشرة وينتقل لمدير الدائرة.
3. **مدير الدائرة** (تبع مدير المشروع تنظيمياً) يراجع ويوافق → ينتقل لمدير الرقابة العامة.
4. **مدير الرقابة العامة** يستلم، يحدد طريقة/مرحلة المراقبة المبدئية، **يعيّن مراقباً محدداً** من فريقه.
   - **عند لحظة تعيين المراقب تحديداً (لا قبلها، لا مجرد الاستلام) يتولّد النشاط الأساسي (primary)** في جدول النشاطات، مربوطاً بالمشروع ومُسنَداً للمراقب مباشرة.
   - من هذه اللحظة: **لا علاقة للمنسق أو مدير الدائرة بالنشاط إطلاقاً**.

**نشاط لا مصدره مشروع** (خارجي مباشر، لاحقاً محضر): **يتخطّى السلسلة 1 بالكامل**، يدخل مباشرة بالسلسلة 2.

### السلسلة 2: المراقبة (موحّدة لكل أنواع الأنشطة)

1. المراقب يفتح النشاط المُسنَد له.
   - إن كان من مشروع: يعبّئ بياناته الميدانية + **عموده هو فقط** بقائمة التحقق (لا يرى عمود/هوية المنسق) + ملاحظات وتوصيات ميدانية.
   - نسبة عمود المراقب الإجمالية = **قيمة التنفيذ في KPI** (يمكن صفراً مؤقتاً).
   - إن كان خارجياً مباشراً: يعبّئ حقول النشاط مباشرة بلا قائمة تحقق مشروع.
2. المراقب يؤكد الانتهاء → يعود لمدير الرقابة العامة.
3. مدير الرقابة يتحقق، يضبط القيم اليدوية المتبقية (جودة/إغلاق/خصم) إن لزم، **يؤكد "اكتمال المرور"** — مستقل تماماً عن اكتمال البيانات أو مستوى KPI.

### الرفض (داخل أي من السلسلتين)

- صلاحية الرفض = **ability مرنة** (الحامل الافتراضي: مدير الرقابة العامة، قابلة للمنح لغيره لاحقاً بلا تغيير بنيوي).
- عند الرفض: **إلزامي** سبب الرفض (نص) + من رفض + التاريخ + **مسؤولية النقص** (من قائمة قصيرة).
- القرار بعد الرفض (إرجاع للمنسق للتصحيح، أو تمرير/حفظ كما هو رغم النقص) **متروك بالكامل لتقدير الجهة الرافضة** — لا مسار إلزامي واحد.

### مخطط التسلسل الكامل

```
[مدير المشروع] ينشئ + يحدد المنسق (قد يكون نفسه)
   ↓
[المنسق] (أو مدير المشروع نيابة عنه) يعبّئ عموده  ── يُتخطّى لو منسق=مدير المشروع ──
   ↓
[مدير الدائرة] يوافق (أو يرفض بسبب + مسؤولية نقص)
   ↓
[مدير الرقابة العامة] يحدد طريقة/مرحلة المراقبة → يعيّن مراقباً
   ↓  🔶 هنا يتولّد النشاط الأساسي 🔶
[المراقب] يعبّئ عموده (منفصل عن رؤية عمود المنسق) + ملاحظات/توصيات
   ↓  نسبة المراقب → قيمة التنفيذ في KPI
[مدير الرقابة العامة] يتحقق، يضبط القيم اليدوية، يؤكد "اكتمال المرور"
   ↓
✅ مكتمل

[نشاط خارجي مباشر] ──▶ [مدير الرقابة العامة] يعيّن مراقباً مباشرة ──▶ (نفس مسار المراقب أعلاه فصاعداً)
```

## 7. نقاط لم تُحسم بعد بشكل نهائي (لا تفترض قراراً — صمّم بافتراض مرن قابل للتعديل)

| # | النقطة | الافتراض المؤقت للعمل |
|---|---|---|
| 1 | صيغة رقم المشروع الدقيقة (رقم صافٍ مؤكَّد، لكن آلية التوليد غير محسومة) | رقم تسلسلي تلقائي، قابل للتعديل اليدوي، مع فرض uniqueness |
| 2 | القائمة الكاملة لقيم "طريقة المراقبة" و"مرحلة المراقبة" (موجود بـ Excel قيمة واحدة فقط لكل منهما) | خزّنهما بـ `constants`، صمّم الواجهة لتسمح بإضافة قيم بسهولة لاحقاً |
| 3 | فئات "مسؤولية النقص" الكاملة عند الرفض، وحدود منح صلاحية الرفض لغير مدير الرقابة | قائمة قصيرة قابلة للتوسيع (منسق/مدير مشروع/غير محدد)، ability مستقلة قابلة للمنح لاحقاً |
| 4 | تفاصيل تحويل ملاحظة ميدانية لنشاط تابع (زر، من يملك الصلاحية، نسخ بيانات أم فورم فارغ) | لا تبنِ الواجهة الآن — فقط تأكد البنية (1:N + activity_role) تدعمه |
| 5 | تفاصيل محضر الاجتماع الكامل (أي جداول يُنشئ، كيف يتحول لنشاط) | مؤجَّل بالكامل — لا تخطط له الآن |
| 6 | هل حقلا "نسبة/حالة الجاهزية" بجدول النشاطات نفس نسبة جاهزية المشروع أم مستقلان | اعتبرهما انعكاساً مباشراً (computed/synced) من نسبة جاهزية المراقب بـ `projects` |
| 7 | بنود "الموارد البشرية" بقائمة التحقق — هل تكليف فعلي أم بيانات توثيقية فقط | بيانات توثيقية نصية فقط، بلا منطق تكليف/إشعار |
| 8 | أولوية التقارير التحليلية بعد إنجاز هذه المرحلة | غير مطلوب الآن إطلاقاً |

## 8. قواعد عامة صارمة (لا استثناء)

- **لا تُنشئ** جدول `roles` تقليدي — النمط الموجود abilities array بملف PHP.
- **لا تخلط** بين `activity_logs` (audit trail عام) وجدول النشاطات الرقابية الجديد — اسمان مختلفان تماماً.
- **لا تبني** استيراد Excel — ملغى من النطاق.
- **لا تبني** محضر الاجتماع أو تقرير القسم أو أي تقرير تحليلي في هذه المرحلة — فقط الأساس + النشاطات + المشروع.
- كل حقل "شخص" = FK لـ `people`، أبداً نص حر.
- كل حقول التقييم الأربعة = أرقام مخزَّنة، التسمية مشتقة للعرض فقط.
- النشاط يُسمح أن يُحفظ **ناقصاً** دائماً — حقل التحقق للعرض فقط، لا يمنع الحفظ أبداً.
- الفورمات = أقسام منظمة عادية، **ليست** أسلوب Wizard خطوة بخطوة.
