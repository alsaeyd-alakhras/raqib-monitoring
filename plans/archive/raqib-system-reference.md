# raqib — المرجع الموحّد النهائي

> ملف واحد يلغي كل ملفات التخطيط السابقة. مرجع النظام كما استقر بعد كل النقاشات وردود العميل.
> Stack: Laravel 11 + Blade + Bootstrap + MySQL · عربي RTL · أرقام إنجليزية.

---

## القسم 1 — الفكرة والنطاق

نظام رقابة وتقييم أداء (M&E/KPI) لجمعية خيرية، محوّل من Excel (12 ورقة) إلى تطبيق ويب.
مراقب ميداني يسجّل نشاطاً رقابياً، النظام يحسب KPI بمعادلة مرجّحة ثابتة.

**النطاق الحالي (بالترتيب):** الأساس (DB + ثوابت + أشخاص + أدوار) ← جدول النشاطات ← المشروع (جدول + فورم + مراحل).
**مؤجَّل:** محضر الاجتماع، تقرير القسم، كل التقارير التحليلية.
**ملغى:** استيراد Excel — إدخال يدوي نظيف.

## القسم 2 — الكود الموجود مسبقاً (احترمه، لا تصطدم به)

| العنصر | الحالة | التعامل |
|---|---|---|
| `users` | موجود | يُربط بـ `people` |
| `constants` (key/value json) | موجود | خزّن فيه: السلالم، طريقة/مرحلة المراقبة، خيارات checklist، أنواع النشاط |

| `role_user`, `currency` | موجود | غير مستخدمين في raqib |
| Auth + Policies | جاهز | ابنِ عليه |

لا Git — يُدار يدوياً.

## القسم 3 — الأدوار (النقطة الحرجة)

**مفهومان منفصلان لا يُخلطان:**

### أ. Abilities = صلاحيات النظام التقنية
موجودة، array بملف PHP. view/create/edit/delete. **ليست أدواراً.** تبقى منفصلة.

### ب. الأدوار الوظيفية = صفة واحدة ثابتة على الشخص
كل شخص له **دور واحد فقط**، مخزَّن كـ صفة على `people` (وليس جدول أدوار منفصلاً ولا داخل abilities):

| الدور | ملاحظة |
|---|---|
| `project_manager` | ينشئ المشاريع · **تابع لدائرة** |
| `coordinator` | منسق |
| `department_manager` | **يدير دائرة معيّنة** |
| `monitoring_director` | مدير الرقابة العامة (مؤسسي) |
| `monitor` | تابع لدائرة الرقابة العامة |
| `general_management` | يرى كل شيء، خارج سلسلة الموافقات |
| `admin` | يمنح الأدوار، يدير الثوابت |

**الانتماء التنظيمي (الدائرة) صفة على الشخص أيضاً** — ضروري لأن المراحل تعتمد عليه (مشروع مدير المشروع يذهب لمدير دائرته تحديداً).

**منسق المشروع = حقل على `projects`** يشير لـ `people`، ليس دوراً عاماً. مدير المشروع يعيّنه (قد يعيّن نفسه).

**تعبئة نيابة:** مدير المشروع يستطيع ملء حقول عمود المنسق في الفورم بدلاً عنه — صلاحية تعبئة فقط، **ليست دوراً ثانياً**. الدور يبقى واحداً.

## القسم 4 — المعمارية: فصل المصدر عن النشاط

```
مصادر (جداول مستقلة، حقول مختلفة):
  projects ──┐
  meetings (مؤجَّل) ──┤ كل مصدر "يولّد" نشاط/أنشطة
  departments (مؤجَّل) ──┘
                    ↓
  monitoring_activities (النشاطات) ← الكيان المركزي المُراقَب
     • يمرّ بمراحل المراقبة (ليس المصدر)
     • مربوط بمصدره: source_type + source_reference
```

**نشاط خارجي مباشر** (سيارة، مبنى): يُدخل مباشرة في جدول النشاطات، `source_type=external`، بلا `source_reference`.

**علاقة المشروع بالنشاط = 1:N:**
- نشاط **أساسي (primary)** واحد إلزامي، يتولّد تلقائياً عند تعيين المراقب.
- أنشطة **تابعة (secondary)** اختيارية، تُنشأ يدوياً لاحقاً (مؤجَّل التنفيذ، لكن البنية تدعمه: `source_reference` بلا unique + حقل `activity_role`).

## القسم 5 — الجداول

### monitoring_activities
هوية: رمز النشاط (تلقائي، بادئة حسب النوع، قابل للتعديل، **unique**) · source_type · source_reference (بلا unique) · activity_role (primary/secondary).
هرم: center → department → section (dependent). الأطراف: مسؤول النشاط (FK people)، المراقب (FK people له user).
زمن: date, time · (day, month, year محسوبة).
تصنيف: نوع النشاط (constant) · اسم/رقم المشروع · الممول (FK funders).
بيانات مشروع (كلها اختيارية، تظهر لو source=project): مدير المشروع، المستفيدون، المناطق، الميزانية، نسبة/حالة الجاهزية، التوصيات.
محتوى: الموضوع، الملاحظة، مشكلة ميدانية (bool)، الإجراء.
تقييم (**أرقام دائماً، التسمية مشتقة للعرض**): تنفيذ (من نسبة المراقب) · جودة · إغلاق · خصم (الثلاثة إدخال يدوي من مدير الرقابة/الإدارة العامة).
محسوب: KPI = (تنفيذ×0.4)+(جودة×0.3)+(إغلاق×0.3)−خصم · التصنيف · ✓تحقق (**عرض فقط، لا يمنع الحفظ**).
مراقبة: طريقة/مرحلة المراقبة (مدير الرقابة يحددهما) · workflow status · **اكتمال المرور** (مستقل عن اكتمال البيانات).

### projects
بيانات المشروع: اسم، رقم (صافٍ، unique)، نوع، الممول، مندوب مشتريات، مدير المشروع (FK people)، **coordinator_id (FK people)**، تواريخ، موقع.
تنفيذ: المستفيدون، المناطق، المدة، الميزانية.
مراقب ميداني: اسمه، طريقة/مرحلة، التاريخ.
قائمة التحقق: **ديناميكية** (مجموعات + بنود، يتحكم بها admin فقط). كل بند عمودان (منسق + مراقب). خيارات: جاهز/غير جاهز/غير مطلوب. **"غير مطلوب" يُستثنى من الحساب.**
ملخص الجاهزية (محسوب): نسبة لكل مجموعة → إجمالي. **نسبة المراقب = قيمة التنفيذ في KPI.** نسبة المنسق توثيقية.
ملاحظات وتوصيات المراقب (قوائم نصية).
حالة المشروع (workflow) + حقول الرفض: سبب (إلزامي)، من رفض، التاريخ، **مسؤولية النقص** (قائمة قصيرة قابلة للتوسيع).
**المراقب لا يرى بيانات/هوية المنسق ولا عموده** — الواجهة/الاستعلام تُخفيها.

### people
موحّد. الاسم + **role** (صفة واحدة) + **department_id** (الانتماء) + user_id (nullable). أي اختيار شخص = FK من هذا الجدول، لا نص حر.

### الهرم التنظيمي
centers (2) → departments (~6-9) → sections (~66). كل مستوى تابع لواحد أعلى فقط.

### constants (الموجود)
السلالم الأربعة (رقم←تسمية):
- تنفيذ: ممتاز100/جيد80/مقبول60/خطر40
- جودة: ممتاز100/جيد85/مقبول70/ضعيف50
- إغلاق: مكتمل100/جزئي60/معلّق30/مفتوح0
- خصم: لا0/تأخير-5/عجز-10/جودة-15/امتثال-20/مخالفة-25
تصنيف KPI: ≥98 ممتاز جداً · ≥90 ممتاز · ≥75 جيد · ≥60 مقبول · ≥40 ضعيف · <40 خطر شديد
خيارات checklist: جاهز/غير جاهز/غير مطلوب.
أنواع النشاط (10)، طريقة/مرحلة المراقبة (قيم Excel نقطة بداية).

### funders
الاسم فقط حالياً.

## القسم 6 — المراحل (Workflows)

**سلسلتان متعاقبتان منفصلتان.**

### سلسلة 1: تحضير/اعتماد المشروع (إدارية)
1. `project_manager` ينشئ المشروع، يعيّن المنسق (قد يكون نفسه).
2. المنسق يعبّئ عموده — أو مدير المشروع نيابةً (تعبئة نيابة). لو مدير المشروع=المنسق، تُتخطّى خطوة الموافقة.
3. `department_manager` (تبع دائرة مدير المشروع) يوافق/يرفض.
4. `monitoring_director` يستلم، يحدد طريقة/مرحلة، **يعيّن مراقباً ← عند التعيين يتولّد النشاط الأساسي**.

نشاط خارجي/محضر: يتخطّى السلسلة 1، يدخل السلسلة 2 مباشرة.

### سلسلة 2: المراقبة (موحّدة)
5. `monitor` يعبّئ عموده (لا يرى المنسق) + ملاحظات. نسبة المراقب → قيمة التنفيذ.
6. `monitoring_director` يتحقق، يضبط القيم اليدوية، يؤكد **اكتمال المرور**.

**الرفض:** ability مرنة (افتراضياً monitoring_director). إلزامي: سبب + من + تاريخ + مسؤولية نقص. القرار بعده (إرجاع أو تمرير) بتقدير الرافض.

## القسم 7 — نقاط معلّقة (صمّم مرناً، لا تفترض نهائياً)

1. صيغة رقم المشروع → تسلسلي auto قابل للتعديل + unique.
2. قوائم طريقة/مرحلة المراقبة الكاملة → في constants، قابلة للإضافة.
3. فئات مسؤولية النقص + حدود صلاحية الرفض → قائمة قصيرة + ability مستقلة.
4. تحويل ملاحظة → نشاط تابع → لا تبنِ الواجهة الآن، البنية تدعمه.
5. محضر الاجتماع كاملاً → مؤجَّل.
6. نسبة/حالة الجاهزية بالنشاط = انعكاس من projects (synced).
7. بنود الموارد البشرية بالتحقق = بيانات نصية، لا تكليف فعلي.

## القسم 8 — قواعد صارمة

- لا جدول roles تقليدي — الدور صفة على `people`.
- لا تخلط الأدوار الوظيفية مع abilities.
- لا تخلط `activity_logs` (موجود) مع جدول النشاطات.
- لا استيراد Excel. لا محضر/تقرير قسم/تقارير تحليلية الآن.
- كل حقل شخص = FK لـ people. كل تقييم = رقم، التسمية عرض.
- النشاط يُحفظ ناقصاً دائماً — التحقق عرض فقط.
- الفورمات = أقسام منظمة، ليست Wizard.
